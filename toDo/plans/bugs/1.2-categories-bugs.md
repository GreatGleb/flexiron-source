## БАГ-01 — Дублирование полей в inheritedFields

**Файл:** `src/services/mocks/categories.ts`  
**Функции:** `mockCreateCategory`, `mockPatchCategory`  
**Серьёзность:** Высокая — данные в STORE были некорректными

### Проблема

При вычислении `inheritedFields` для новой/переприкреплённой категории использовался рекурсивный цикл вверх по цепочке предков. На каждой итерации добавлялись `inheritedFields + fields` родителя. Поскольку `inheritedFields` каждого родителя уже содержит накопленную цепочку от его предков, поля верхнего уровня попадали в результат несколько раз.

**Пример (категория «Алюминиевые листы», parentId = «Листы»):**

```
Металл.inheritedFields = []
Металл.fields = [f-1-1 (Марка стали), f-1-2 (ГОСТ)]

Листы.inheritedFields = [f-1-1, f-1-2]   ← уже накоплено от Металла
Листы.fields = [f-2-1 (Толщина), f-2-2 (Тип), f-2-3 (Размер)]
```

Цикл при создании дочерней категории «Алюминиевые листы»:
- Итер 1 (Листы): добавляет [f-1-1, f-1-2, f-2-1, f-2-2, f-2-3]
- Итер 2 (Металл): добавляет [f-1-1, f-1-2]
- Итог: **[f-1-1, f-1-2, f-1-1, f-1-2, f-2-1, f-2-2, f-2-3]** ← дубли!

### Исправление

Прямой родитель уже хранит корректную накопленную цепочку в своём поле `inheritedFields`. Достаточно взять у него `inheritedFields + fields`:

```ts
// было (рекурсивный цикл — баг):
let currentParentId = parentId
const chain: CategoryField[][] = []
while (currentParentId) {
  const parent = STORE.find((c) => c.id === currentParentId)
  if (!parent) break
  chain.unshift([...parent.inheritedFields, ...parent.fields])
  currentParentId = parent.parentId
}
chain.forEach((fields) => inheritedFields.push(...fields))

// стало (один вызов к прямому родителю — правильно):
const parent = STORE.find((c) => c.id === parentId)
const inheritedFields = parent ? [...parent.inheritedFields, ...parent.fields] : []
```

## БАГ-06 — mockGetCategories не сортирует depth-first

**Файл:** `src/services/mocks/categories.ts`  
**Функция:** `mockGetCategories`  
**Серьёзность:** Средняя — после создания новых категорий порядок в списке ломался

### Проблема

Контракт требует: "Сортировка: корневые категории по `name ASC`, дочерние после своего родителя (depth-first)". Mock возвращал `STORE` в порядке вставки. Новые категории (`STORE.push`) уходили в конец, разрывая дерево.

**Пример:** Создали дочернюю «Медь» для «Металл» → в списке она появлялась после «Оборудование», а не сразу после «Металл» и его прочих детей.

### Исправление

```ts
function depthFirstSort(items: Category[]): Category[] {
  const result: Category[] = []
  function visit(parentId: string | null) {
    items
      .filter((c) => c.parentId === parentId)
      .sort((a, b) => a.name.localeCompare(b.name, 'ru'))
      .forEach((c) => { result.push(c); visit(c.id) })
  }
  visit(null)
  return result
}
```

`mockGetCategories` теперь применяет `depthFirstSort(filtered)` перед пагинацией.

## БАГ-07 — mockPutCategoryFields и mockPatchCategory не каскадируют inheritedFields потомкам

**Файл:** `src/services/mocks/categories.ts`  
**Функции:** `mockPutCategoryFields`, `mockPatchCategory`  
**Серьёзность:** Высокая — после изменения полей родителя, дочерние карточки показывали устаревший список унаследованных полей

### Проблема

- `mockPutCategoryFields(cat-1)` обновляет `cat-1.fields`, но `cat-2.inheritedFields` и `cat-3.inheritedFields` (потомки) остаются старыми
- `mockPatchCategory(cat-2, { parentId: 'cat-5' })` обновляет `cat-2.inheritedFields`, но `cat-3.inheritedFields` (дочерний «Алюминиевые листы») не обновляется

**Пример:** Добавили поле «Сертификат» к «Металл» → GET «Листы» всё ещё не видит «Сертификат» в `inheritedFields`.

### Исправление

```ts
function cascadeInheritedFields(parentId: string): void {
  const parent = STORE.find((c) => c.id === parentId)
  if (!parent) return
  STORE.filter((c) => c.parentId === parentId).forEach((child) => {
    child.inheritedFields = [...parent.inheritedFields, ...parent.fields]
    cascadeInheritedFields(child.id)
  })
}
```

Вызывается:
- В конце `mockPutCategoryFields` — `cascadeInheritedFields(id)`
- После обновления `inheritedFields` в `mockPatchCategory` — `cascadeInheritedFields(cat.id)`

## БАГ-13 — depthFirstSort ломает поиск по дочерним категориям

**Файл:** `src/services/mocks/categories.ts`  
**Функция:** `mockGetCategories`  
**Серьёзность:** Средняя — поиск возвращает пустой список при запросе non-root категорий

**Пример:** поиск `"листы"` → filtered = `[Листы, Алюминиевые листы]` → `depthFirstSort` → `[]`.

### Исправление

```ts
const sorted = filters.search
  ? filtered.slice().sort((a, b) => a.name.localeCompare(b.name, 'ru'))
  : depthFirstSort(STORE)
```

При поиске — плоская сортировка по имени. Иерархический порядок нужен только для полного списка без фильтра.

## БАГ-23 — SearchInput: иконка поиска вылезала за пределы поля

**Файл:** `src/components/admin/ui/SearchInput.vue`  
**Серьёзность:** Средняя — иконка поиска визуально некорректна: появлялась только когда сайдбар закрыт, и вылезала за границу поля

### Проблема

`SearchInput.vue` использовал CSS-класс `checkbox-list-search` для корневого элемента. Этот класс определён в `_checkbox-list.css` и задаёт:

```css
.checkbox-list-search { position: relative; }
.checkbox-list-search .search-icon { position: absolute; left: 12px; top: 50%; transform: translateY(-50%); }
.checkbox-list-search .search-input { padding-left: 36px; }
```

Но `SearchInput.vue` **не импортировал** `_checkbox-list.css`. Стили работали только случайно — когда на той же странице использовался `CheckboxList.vue`, который подгружал `_checkbox-list.css` вместе с собой.

На странице категорий `CheckboxList` не используется → `_checkbox-list.css` не загружается → `.checkbox-list-search` не имеет `position: relative` → иконка позиционируется как static-элемент и торчит за пределами поля.

### Исправление

1. Создан `src/styles/admin/components/_search-input.css` с классом `.search-input-wrap`:

```css
.search-input-wrap { position: relative; }
.search-input-wrap .search-icon { position: absolute; left: 12px; top: 50%; transform: translateY(-50%); pointer-events: none; }
.search-input-wrap .search-input { padding-left: 36px; font-size: 13px; width: 100%; box-sizing: border-box; }
```

2. `SearchInput.vue` импортирует `_search-input.css` и использует класс `search-input-wrap` — стили загружаются всегда, независимо от других компонентов на странице.

3. `_checkbox-list.css` оставлен без изменений — у `CheckboxList.vue` собственная разметка поиска со своим классом `checkbox-list-search`.

### Правило на будущее

Компонент **обязан сам импортировать** свои стили. Нельзя полагаться на то, что нужные классы загрузит другой компонент на той же странице.

## PAT-02 — Ссылки-навигация на другие страницы раздела

**Файл:** `src/views/admin/products/ProductsPage.vue`  
**Серьёзность:** Низкая — паттерн нарушен (кастомные стили вместо готовых компонентов)

### Проблема

При добавлении ссылки «Категории» на страницу товаров были созданы кастомные CSS-классы `.products-section-card` и `.products-section-links` с нуля. Параллельно был использован `config-section-card` — семантически неверный класс (предназначен для drag-drop секций конфига, не для навигации).

### Правило

Ссылки-навигация на другие страницы раздела делаются через:

```html
<!-- обёртка в хедере страницы -->
<div class="entity-action-bar no-margin pos-static">
  <router-link :to="{ name: 'admin-target' }" class="btn btn-secondary">
    <SvgIcon name="icon-name" :width="18" :height="18" />
    <span>{{ t('section.title') }}</span>
  </router-link>
</div>
```

- **`router-link`** — не `<a href>`, всегда через именованный маршрут
- **`btn btn-secondary`** — готовый стиль кнопки из `_buttons.css`, не кастомный CSS
- **`entity-action-bar no-margin pos-static`** — готовый flex-контейнер для кнопок в хедере из `_entity-card-layout.css`; `no-margin` убирает отступ, `pos-static` убирает sticky
- **Не создавать** новых CSS-классов для навигационных ссылок — все стили уже есть в core

## БАГ-25 — Поиск пропадает и теряет фокус при вводе символа

**Файл:** `src/composables/useCategories.ts`  
**Серьёзность:** Высокая — UX полностью сломан: поиск исчезает на время запроса, фокус сбрасывается

### Проблема

`watch(filters, load, { deep: true })` вызывает `load()` при каждом изменении поиска. `load()` ставит `loading = true` → GlassPanel применяет `.glass-panel.loading .panel-body { display: none }` → поиск (находящийся в `panel-body`) скрывается. Браузер сбрасывает фокус с элемента у которого `display: none`. При завершении запроса panel-body снова показывается, но фокус потерян.

### Исправление

Скелетон показывается только при **первой** загрузке. После инициализации — тихое обновление без скрытия panel-body:

```ts
let initialized = false

async function load() {
  if (!initialized) loading.value = true  // скелетон только первый раз
  try {
    const res = await getCategories(filters)
    items.value = res.items
    initialized = true
  } finally {
    loading.value = false
  }
}
```

### Правило на будущее

Если поиск/фильтры находятся **внутри** `GlassPanel` — watcher фильтров не должен включать `loading = true`. Использовать `initialized` флаг или отдельный `filtering` ref без связи со скелетоном.

## PAT-04 — Обновление visual snapshot baseline после UI-изменений

**Серьёзность:** Низкая — процессный паттерн

### Правило

После любого изменения которое затрагивает визуальный вид элемента с snapshot-тестом — **сразу обновить baseline**:

```bash
npx playwright test [spec-file] -g "[test-name]" --update-snapshots --workers=1
```

Затем открыть обновлённый PNG (`tests/e2e/...-snapshots/*.png`) и визуально убедиться что он корректен.

**Когда обязательно обновлять:**
- Добавлен/изменён CSS (margin, padding, color, font-size) на элементе с snapshot
- Изменён текст кнопки/заголовка (например "Discard" → "Discard changes")
- Добавлен или убран элемент внутри snapshot-зоны

**Не обновлять автоматически** — сначала убедиться что изменение намеренное, а не регрессия.

**Примеры из этой страницы:**
- `category-card-own-fields` — обновлён после `margin-top: 12px` на кнопке Add field
- `category-card-header` — обновлён после переименования "Discard" → "Discard changes"
- `categories-filters` — обновлён после исправления иконки поиска (`SearchInput`)
