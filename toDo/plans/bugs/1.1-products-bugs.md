## ✅ БАГ-01 — Неиспользуемые импорты типов вызывали TypeScript TS6196

**Файлы:**
- `src/services/mocks/products.ts` — `LinkedSupplier` импортирован, не используется
- `src/services/productsService.ts` — `LinkedSupplier` импортирован, не используется
- `src/composables/useProductCard.ts` — `PriceUnit` импортирован, не используется
- `src/views/admin/products/ProductCardPage.vue` — `SvgIcon` импортирован, не используется

**Серьёзность:** Средняя — `npm run typecheck` падал

### Проблема

`LinkedSupplier` использовался только внутри `Product.linkedSuppliers` — тип массива элементов. TypeScript выводит его автоматически из интерфейса `Product`, явный импорт не нужен. Аналогично `PriceUnit` в `useProductCard.ts` — используется только как часть `Product`, не как самостоятельный тип. `SvgIcon` был добавлен в imports ProductCardPage.vue, но потом не использован в шаблоне (нет `<SvgIcon>` тегов в карточке).

### Исправление

Удалены из списков импортов:
- `LinkedSupplier` из `products.ts` и `productsService.ts`
- `PriceUnit` из `useProductCard.ts`
- `SvgIcon` из `ProductCardPage.vue`

### Правило на будущее

Перед написанием импортов — убедиться что тип используется явно в подписях или `ref<T>()`, а не только косвенно через другой интерфейс. Запускать `typecheck` после каждого Промпта, не только в конце.

## ✅ БАГ-02 — CustomSelect не принимает `string | null` в v-model

**Файл:** `src/views/admin/products/ProductsPage.vue`  
**Серьёзность:** Высокая — TypeScript ошибка + runtime несоответствие

### Проблема

`CustomSelect` ожидает `modelValue: string` (SelectOption.value всегда string). Поля `filters.categoryId: string | null` и `newProduct.categoryId: string | null` несовместимы — TypeScript-ошибка при прямой привязке `v-model="filters.categoryId"`.

Кроме того, `categoryOptions` первым элементом имел `{ value: null }` — `CustomSelect` сравнивает `value === modelValue`, и null никогда не совпадёт со строкой.

### Исправление

Computed-адаптеры для каждого поля `string | null`:

```ts
const categoryFilterStr = computed({
  get: () => filters.categoryId ?? '',
  set: (v: string) => { filters.categoryId = v || null },
})

const newProductCategoryStr = computed({
  get: () => newProduct.value.categoryId ?? '',
  set: (v: string) => { newProduct.value.categoryId = v || null },
})
```

`categoryOptions` первым элементом: `{ value: '', label: t('products.filter_category_all') }` (пустая строка, не null).

### Правило на будущее

`CustomSelect` принимает **только `string`** в v-model и в `SelectOption.value`. Для любого поля `string | null`:
1. Создать computed-адаптер `null ↔ ''`
2. Опция «Все/пусто» → `value: ''`, не `value: null`

## ✅ БАГ-05 — `v-model.number` ставит NaN при очистке поля — null не восстанавливается

**Файл:** `src/composables/useProductCard.ts`  
**Серьёзность:** Средняя — очистка поля цены/миностатка оставляла `NaN` в данных вместо `null`

### Проблема

`v-model.number` в Vue 3 вызывает `parseFloat("")` при пустом input → результат `NaN`. Далее:

- `JSON.stringify(NaN) === "null"` — `useDirtyCheck` **не замечает** изменение, если исходное значение тоже было `null`. Это корректно.
- Но если исходное значение было, например, `15.5` и пользователь очистил поле → dirty.diff() возвращает `{ price: NaN }` → `mockPatchProduct` сохраняет `NaN` в STORE → после reload поле показывает `NaN`.

```ts
// до исправления — NaN попадал в patchProduct:
form.value.price = NaN   // ← после очистки поля
await patchProduct(id, { price: NaN })  // ← STORE.price = NaN
```

### Исправление

Добавлен watcher в `useProductCard.ts` после `useDirtyCheck(form)`:

```ts
watch(
  form,
  (val) => {
    if (Number.isNaN(val.price as unknown)) form.value.price = null
    if (Number.isNaN(val.minStock as unknown)) form.value.minStock = null
  },
  { deep: true },
)
```

Как только `v-model.number` ставит NaN — watcher немедленно заменяет на `null`. `useDirtyCheck` видит `null` (валидное значение), diff корректен.

### Правило на будущее

Для **nullable number полей** (`number | null`) с `v-model.number` — обязателен watcher-нормализатор NaN → null. Альтернатива: использовать `v-model` (без `.number`) + computed с явным `parseFloat` и fallback на `null`.

## ✅ БАГ-06 — `DropZone` — не v-model компонент, требует `hint: string`

**Файл:** `src/views/admin/products/ProductCardPage.vue`  
**Серьёзность:** Высокая — компонент DropZone при неверном использовании не рендерил hint и не вызывал upload

### Проблема

Первоначальный черновик использовал `v-model` на `DropZone`:

```html
<!-- неверно: -->
<DropZone v-model="fieldValues[fv.fieldId]" />
```

`DropZone` не имеет `modelValue` prop — `v-model` молча игнорировался. Также `hint` — required prop — был пропущен, что давало runtime warning.

### Исправление

```html
<DropZone
  hint="PDF / image"
  @uploaded="(files) => ((fieldValues as Record<string, unknown>)[fv.fieldId] = files[0]?.fileId ?? null)"
/>
```

`@uploaded` эмитирует `UploadedFile[]`. Берём первый файл, сохраняем `fileId` в `fieldValues`.

### Правило на будущее

`DropZone` — **не v-model**. Контракт:
- `hint: string` — обязательный prop, текст подсказки
- `@uploaded="(files: UploadedFile[]) => ..."` — событие после загрузки
- Нет `modelValue`, нет `v-model`

## PAT-01 — `useDirtyCheck` использует `capture()`, а не `track()`

**Файл:** `src/composables/useProductCard.ts` (и любые будущие карточки)  
**Серьёзность:** Средняя — вызов несуществующего метода → runtime error «is not a function»

### Проблема

В плане (`1.1-products-plan.md`) и при первом написании composable был использован метод `track()`:

```ts
// план изначально написал:
dirty.track()  // ← не существует
```

Реальный API `useDirtyCheck`:

```ts
const { isDirty, capture, reset, diff } = useDirtyCheck(form)
// capture() — зафиксировать текущее состояние как baseline
// diff()    — shallow diff: только изменённые ключи
// reset()   — алиас capture()
// isDirty   — computed<boolean>
```

### Исправление

`capture()` вызывается сразу после заполнения `form.value` данными из API:

```ts
form.value = { name: data.name, sku: data.sku, ... }
dirty.capture()  // ← правильный метод
```

### Правило на будущее

`useDirtyCheck` публичный API:
| Метод/prop | Описание |
|---|---|
| `isDirty` | `Ref<boolean>` — есть ли изменения |
| `capture()` | зафиксировать baseline (вызывать после load) |
| `diff()` | возвращает `Partial<T>` только изменённых ключей |
| `reset()` | алиас `capture()` — сброс к baseline |

Метода `track()` не существует.

## ✅ БАГ-09 — `<span class="glass-input">` для read-only поля — width:100% не работает

**Файл:** `src/views/admin/products/ProductCardPage.vue`  
**Серьёзность:** Средняя — поле «Категория» уже других полей (span inline, не block)

### Проблема

`.glass-input` задаёт `width: 100%`, но не задаёт `display`. На `<input>` и `<textarea>` это работает (они block/inline-block по умолчанию). На `<span>` — `display: inline`, поэтому `width: 100%` не применяется: поле отображается только по ширине текста.

```html
<!-- неверно: -->
<span class="glass-input">{{ product?.categoryName ?? '—' }}</span>
```

### Исправление

```html
<input
  :value="product?.categoryName ?? '—'"
  class="glass-input"
  type="text"
  readonly
  data-test="field-category"
/>
```

`<input readonly>` визуально идентичен редактируемому полю, но не принимает ввод.

### Правило на будущее

Read-only отображение значения в форме → `<input :value="..." readonly class="glass-input">`, не `<span class="glass-input">`. Span — только для текстовых меток вне формы.

## ✅ БАГ-10 — `form.sku` / `form.description` сохранялись как `''` вместо `null`

**Файл:** `src/composables/useProductCard.ts`  
**Серьёзность:** Средняя — после очистки поля список товаров показывает пустую ячейку вместо `—`

### Проблема

`<input v-model="form.sku">` при очистке поля ставит `form.sku = ''`. `useDirtyCheck.diff()` возвращает `{ sku: '' }`. Это уходит в `patchProduct` → STORE хранит `sku: ''`. В списке: `item.sku ?? '—'` — `??` не срабатывает на `''`, ячейка пуста.

### Исправление

В `save()` после `Object.assign(delta, dirty.diff())` добавлена нормализация:

```ts
if (delta.sku === '') delta.sku = null
if (delta.description === '') delta.description = null
```

Нормализация только при сохранении — не мешает вводу (пустая строка — допустимое промежуточное состояние).

### Правило на будущее

После `dirty.diff()` нормализовать nullable string поля: `if (delta.field === '') delta.field = null`. Поля с `?? '—'` в шаблоне не различают `null` и `''` — нужно гарантировать `null` из save.

## БАГ-44 — Процесс: нарушение DRY — создание кода без проверки существующих компонентов

**Файлы:** Любые Vue-страницы и компоненты  
**Серьёзность:** Высокая — ведёт к дублированию кода, несогласованному дизайну, невозможности централизованных правок

### Проблема

Перед созданием нового UI-кода не проводится проверка существующей кодовой базы. Примеры нарушений в этой сессии:
- Создан кастомный `file-chip` (CSS + разметка) хотя `FileItem.vue` уже существовал в `src/components/admin/`
- Добавлено `.btn:disabled` в `_buttons.css` хотя `btn-save` уже управляет своим disabled-видом через `.dirty`-класс — это было видно в `SupplierCardPage.vue` как эталоне

Следствие: дублирование кода, рассогласование дизайна между страницами, потраченное время на откат.

### Правило: обязательная проверка перед созданием

**Перед любым новым UI-элементом (компонент, CSS-класс, блок шаблона):**

1. `Grep` по `src/components/` на похожий функционал по ключевым словам (file, chip, tag, badge, dropzone и т.д.)
2. Просмотр эталонной страницы того же домена (например `SupplierCardPage` → эталон для `ProductCardPage`)
3. Если найдено готовое решение — использовать его, не создавать новое
4. Если нужно обобщить (вынести общую часть в компонент) — делать это явно по протоколу ниже

### Протокол рефакторинга (вынос в компонент)

**Шаг 1 — найти все места использования:**
- Grep по всем файлам: найти где дублируется логика/разметка
- Зафиксировать список файлов

**Шаг 2 — создать/обновить компонент:**
- Компонент должен принимать props для всех вариантов использования
- Не ломать существующий API (обратная совместимость)

**Шаг 3 — обновить все места использования:**
- Заменить дублированный код импортом и использованием компонента
- Все файлы из Шага 1

**Шаг 4 — перечислить в конце промпта:**
- Какие страницы затронуты рефакторингом
- Где требуется ручная проверка в браузере

### Цикл промптов для рефакторинга

**Цикл создания (макс. 5 промптов):**
- Каждый промпт = один конкретный шаг (grep → компонент → обновление → проверка типов)
- СТОП после каждого шага
- Остановка: компонент создан и все места обновлены — или использовано 5 промптов

**Цикл верификации (макс. 5 промптов, можно меньше):**
- Каждый промпт подробно проверяет: типы, шаблон, CSS, поведение на старых и новых страницах
- Если промпт делает вывод «всё ок» — цикл завершается досрочно
- Промпты указывают конкретные файлы, пути и сценарии: старые страницы должны работать как раньше, новые — использовать тот же компонент

### Исправление

Применять протокол при каждом создании нового UI-кода. Проверять `src/components/` перед написанием любого нового CSS-класса или блока разметки.
