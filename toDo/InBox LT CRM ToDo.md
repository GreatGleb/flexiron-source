# План разработки CRM-системы складского учета

**Вводная информация:** 
Ниже представлен детализированный план функций новой CRM и складской системы. Весь функционал спроектирован на базе глубокого анализа действующих рабочих Excel-таблиц (учет металла, генерация обрезков, создание таможенных и локальных счетов). Данный документ описывает эталонный функционал, утвержденный для последующей разработки.

## 1. Раздел: СКЛАД
### 1.1 `page Товары` (Справочник номенклатуры)
**Цель:** Справочная база всех видов продукции (эталонные карточки товаров). Физическое наличие, партии и обрезки здесь не учитываются — это функции страницы "Склад".
- [ ] **Динамические категории:**
  - Создание, редактирование и удаление категорий (напр., "Трубы", "Листы", "Декор. профили").
  - Конструктор свойств: возможность добавить любое свойство (число, строка, список) к конкретной категории. Все товары этой категории получат эти свойства в свою карточку в виде полей для заполнения.
- [ ] **Сводный список (Таблица-каталог):**
  - Общий список всех товаров из всех категорий одновременно.
  - В общей таблице отображаются только базовые/общие характеристики (Название, Базовая цена, Единицы измерения).
- [ ] **Карточка "вида товара":**
  - Заполнение базовых данных и динамических полей (специфичных для выбранной категории).
  - Задание эталонной стоимости (EUR/vnt, EUR/kg, EUR/m).
  - **Привязка Поставщиков:** Внутри карточки товара — редактируемый список конкретных поставщиков данного товара (с их ценами и сроками). Администратор вручную добавляет и удаляет поставщиков. Аналогично — привязка товара к поставщику возможна из карточки поставщика. BCC requests к этому списку не относятся.

### 1.2 `page Склад` (Физическое наличие, партии и обрезки)
**Цель:** Главная рабочая страница. Учет реального физического наличия, движения партий и калькуляция остатков/обрезков.
- [ ] **Учет движения и партий:**
  - Приход (закупка) с привязкой к эталонному "Товару" из справочника. Разделение по партиям.
  - Расход (продажа/списание).
- [ ] **Динамические складские свойства (Гибкость структуры):**
  - Способность создавать динамические поля категории *именно для склада*, которые не имеют отношения к карточке эталонного товара (например, "Состояние: Погнут", "Ячейка/Полка", "Резерв").
  - Это позволяет не "хардкодить" статусы и легко пополнять формы новыми логистическими признаками.
- [ ] **Настраиваемое отображение таблиц (Views):**
  - Два формата обзора: (1) Общий (сводный) список всех остатков с базовыми колонками, и (2) Просмотр остатков конкретной категории.
  - В просмотре по категории становятся видны специфические колонки (оба типа: эталонные из карточки товара и складские).
  - Настройка видимости и порядка колонок менеджером.
- [ ] **Система "Обрезков" (Atraižos):**
  - Автоматизированный механизм отрезания части от целого: система генерирует физический остаток нестандартного размера как отдельный объект, связанный родительским ID.
  - Учет технологических потерь: При отрезании система запрашивает или подставляет «Ширину реза» (например, +3 мм на пропил), чтобы списание со склада соответствовало физическому остатку, а не только чистой длине детали.
  - Калькуляция параметров обрезка (авто-расчет его нового веса, площади и цены пересчитывается программно по формулам родителя).
  - Нестандартные формы (Фотографии): Возможность прикрепить фотографию или чертеж к конкретному физическому остатку сложной формы (после лазера/фрезеровки) для визуальной идентификации.
  - Маркировка и QR-коды: Автоматическая генерация наклейки (Lipdukai) с QR-кодом для каждого обрезка. При сканировании кладовщик видит размеры, марку стали и ссылку на родительскую партию.
  - Интеграция с лазерной резкой (OptiCut): Архитектурный задел для связи склада с программами оптимизации раскроя (чтобы система подсказывала наиболее оптимальный кусок для резки).


### 1.3 `page Услуги` (Справочник нематериальных позиций)
**Цель:** База услуг компании (Резка металла, Доставка, Индивидуальная упаковка, Погрузка).
- [ ] **Карточка услуги:** Название и Тип услуги.
- [ ] **Тарификация:** Настройка базовой эталонной стоимости. Задание Себестоимости (cost) и Наценки (retail price).

## 2. Раздел: CRM
### 2.1 `page Клиенты` (Справочник контрагентов)
**Цель:** Эталонная база субъектов для быстрого подтягивания реквизитов в документы. Как и "Товары", это лишь справочник. Основная работа с клиентами происходит в "Заказах".
- [ ] **Базовая информация (Реквизиты):**
  - Название компании (Pirkėjas), Код компании, PVM / VAT код, Адрес, Телефон, Email.
- [ ] **Динамические свойства клиента:**
  - Гибкий конструктор для создания любых дополнительных полей (например: "Категория: VIP", "Условия отсрочки", "Любимая транспортная компания").
- [ ] **История работы:**
  - Сводка всех заказов и выставленных счетов по данному клиенту.
  - **Аналитика отказов:** Фиксация причины отказа, если клиент запрашивал цену, но не купил (например, "дорого", "нет в наличии").

### 2.2 `page Заказы` (Формирование отгрузок и генерация документов)  
**Цель:** Главная рабочая панель продаж (Сделка). Создание заказов, резервирование товара на складе, авто-калькуляция и генерация всех закрывающих отгрузочных документов.
- [ ] **Создание заказа (Шапка):**
  - Умный поиск и выбор клиента из Справочника (все его реквизиты, PVM, юридические адреса подхватываются и запоминаются для документов).
  - Выбор типа комплекта: Локальный (счета и накладные Литвы) или Экспорт (накладные EN, CMR).
- [ ] **Корзина заказа и связь со Складом:**
  - Сокращение остатков: списывание доступных позиций с `page Склад`.
  - **Механизм "Отрезать":** Если клиент покупает половину листа или часть 6-метровой трубы, Заказ "отрезает" нужный кусок, продает его, а остаток (обрезок нужных размеров) система автоматически возвращает на `Склад`.
  - Резервирование: статус товара временно "заблокирован" (бронь), пока сделка не будет закрыта.
- [ ] **Добавление Услуг в заказ:**
  - Подтягивание услуг из Справочника (Резка, Доставка) прямо в чек к клиенту.
  - **Гибкое ценообразование:** базовая цена тянется из справочника, но для конкретного заказа менеджер может вручную поменять цену услуги (дать скидку или накинуть сверху).
  - Показ маржинальности: система берет Себестоимость услуги из Справочника и выводит итоговую чистую прибыль по заказу.
- [ ] **Финансовая и логистическая автоматизация:**
  - Общий калькулятор габаритов заказа (Итоговый вес (Bendras svoris) всех труб и листов).
  - Системное применение скидок на строку либо на весь Заказ целиком.
  - Авто-расчет стоимости с НДС (su PVM) и базовой (без НДС).
- [ ] **Генератор печатных документов (Invoice & Logistic):**
  - Шаблонизатор на базе ваших Excel файлов (`zag...`). За один клик заказ генерирует предпросмотр нужных вам документов (Proforma-Invoice, CMR, KPO, Сертификаты, Lipdukai).
  - **Ручная корректировка "на лету":** Перед скачиванием или отправкой инвойса менеджер может вручную исправить текст, цены или названия прямо в форме счета (чтобы быстро поправить мелкие нестыковки без возвращения в карточку товара).
- [ ] **Динамические свойства заказа:**
  - Гибкий конструктор дополнительных полей для самого заказа. Позволит в будущем добавить, например, "Номер договора клиента (PO)", "Приоритет сборки", "Хрупкость" или "Имя водителя/курьера".
- [ ] **Кастомная воронка статусов и логика резерва:**
  - Конструктор статусов заказа: возможность создавать любые свои статусы (например, "Сборка", "Проверка", "В пути").
  - Триггеры склада: настройка правил — при каких статусах товар ставится в "Резерв" (бронируется на складе), при каких статусах резерв снимается (отмена заказа), а при каких товар окончательно "Списывается".
  - Управление временем резерва: Настройка «Срока жизни резерва» (например, 3 рабочих дня). По истечении срока система уведомляет менеджера или автоматически снимает бронь, возвращая товар в свободную продажу.

## 3. Раздел: ЗАКУПКИ И ПОСТАВЩИКИ (Supplier & Purchase)
### 3.1 `page Поставщики`
**Цель:** Обособленная CRM-база поставщиков металла с условиями их работы (минимальный заказ для бесплатной доставки, сроки и т.д.).
- [ ] **Профили поставщиков:** Привязка карточки поставщика к товарным категориям рынка металлов, которые он продает.
- [ ] **Алгоритм определения дефицита поставщика (Supply Health):**
  - Автоматический расчет показателя `hasDeficit` на основе множества факторов:
    - **BCC Request History:** Отсутствие ответа на 2+ запроса (+30 баллов), запросы в статусе "pending" более 7 дней (+20 баллов).
    - **Deficit Items:** Товары в статусе Critical/Low, соответствующие категориям поставщика (+25 баллов за 2+ совпадения).
    - **Pricing History:** Резкие падения Stock в последние месяцы (+15 баллов).
    - **Status поставщика:** "Under Review" (+10 баллов), "Suspended" (+40 баллов).
  - Порог: 50+ баллов = поставщик получает метку `hasDeficit`.
  - **Отображение в карточке:** Секция "Supply Health" с визуализацией рисков, списком проблемных позиций и рекомендациями.
  - **Индикатор в списке:** Красная точка рядом с названием компании при `hasDeficit = true`.

### 3.2 `page Потребности` (Low Stock Alerts)
**Цель:** Автоматизированный контроль дефицита товаров на складе.
- [ ] **Установка лимитов:** Задание неснижаемого остатка (порога) для металла.
- [ ] **Автоматический список закупок:** Система сама проверяет складское наличие и формирует список позиций («подсвечивает желтым»), которые пора заказывать (без необходимости проверять листы поштучно).

### 3.3 `Массовые запросы цен` (BCC Messaging)
**Цель:** Автоматизация общения со списком поставщиков для получения актуальных цен на "неизвестные" или проектные позиции.
- [ ] **Скрытая рассылка в 1 клик:** Кнопка в категории металла, формирующая шаблонное e-mail письмо с запросом текущих цен и сроков. Отправка идет сразу всем потенциальным поставщикам металла через скрытую копию (BCC).
- [ ] **История цен:** Сохранение присланных предложений для аналитики закупочных цен.

## 4. Раздел: СИСТЕМНАЯ АРХИТЕКТУРА И НАСТРОЙКИ
### 4.1 Пользовательские роли и права
- [ ] **Матрица доступов:** Разделение ролей на Администратор, Продажи, Склад, Бухгалтерия. Права настраиваются гибко (например, склад не видит розничных/закупочных цен, бухгалтер видит только проведенные отгрузки).

### 4.2 Облачная инфраструктура и Электронный архив
- [ ] **Общая online база данных:** Размещение серверной части на облачном VPS (~30€/год) без установки локальных физических серверов. При этом клиенты скачивают быстрое приложение (клиент на Windows). Это обеспечит единую синхронную работу для всех сотрудников из любой точки мира.
- [ ] **Электронный архив (Paperless):** Сохранение созданных PDF-файлов и фактур прямо внутри истории заказа на сервере (заменяет бумажные архивы).
- [ ] **Масштабируемость и API:**
  - Архитектурный задел для подключения внешнего оборудования (электронные весы, сканеры штрих-кодов) через API для автоматизации приемки и инвентаризации без ручного ввода веса.

### 4.3 Мультиязычность
- [ ] **Локализация:** Системная архитектура строится на английском языке (под капотом на базе React/Python), но с полной поддержкой переводов интерфейса и печатных форм (Русский, Литовский, Английский).

### 4.4 Журнал аудита (Activity Log / Audit Trail)
- [ ] **Автоматическая фиксация:** Регистрация любого действия в системе (создание, редактирование, удаление, смена статуса).
- [ ] **Состав записи:** Система фиксирует «Кто изменил», «Когда (дата/время)», «Какое поле», «Старое значение» → «Новое значение».
- [ ] **История в карточке:** В каждом объекте (Товар, Заказ, Клиент, Партия) доступна вкладка или блок «История изменений» для просмотра хронологии действий именно по этому объекту.
- [ ] **Защита данных:** Логи доступны только для чтения и не могут быть удалены или изменены пользователями.

## 5. Раздел: СИСТЕМА УВЕДОМЛЕНИЙ (Notifications)
**Цель:** Информировать пользователей о важных событиях в системе в режиме реального времени.
- [ ] **Внутренние уведомления (In-app):**
  - Центр уведомлений внутри интерфейса (колокольчик с счётчиком непрочитанных).
  - Каждое уведомление содержит **прямую ссылку** на соответствующий объект в системе (конкретный заказ, партию на складе, карточку клиента и т.д.).
  - Примеры триггеров: "Заказ #123 перешёл в статус «В пути»", "Металл [Лист алюм. 3мм] достиг нижнего лимита остатка", "Пришёл ответ от поставщика".
- [ ] **Push-уведомления:**
  - Поддержка браузерных Push-уведомлений (Web Push / PWA) — уведомления приходят даже когда вкладка закрыта.
- [ ] **Настройка подписки:**
  - Каждый пользователь сам выбирает, на какие события он хочет получать уведомления.

## 6. Раздел: КОНСТРУКТОР СВОЙСТВ (Admin Configuration)
**Цель:** Максимально гибкая, zero-code система настройки структуры любой сущности (Товар, Склад, Клиент, Заказ, Поставщик) администратором — без участия программиста.

### 6.1 Структура полей: Разделы и порядок
- [ ] **Разделы (Groups):** Возможность объединять несколько свойств в именованный раздел (например: раздел "Физические характеристики" содержит поля "Длина", "Ширина", "Толщина", "Вес").
- [ ] **Drag-and-drop порядок:** Администратор может перетаскиванием (drag-and-drop) менять порядок как отдельных полей внутри раздела, так и порядок самих разделов между собой. Итоговый порядок сохраняется и отображается в карточке объекта у всех пользователей.

6.2 Типы свойств (Field Types)
- [ ] **Простые и системные поля:**
  - Текст/Число/Логическое: Базовые типы для простых атрибутов.
  - Электронная почта (Email): Поле с автоматической валидацией формата (наличие `@`, корректность домена). Позволяет в один клик инициировать отправку письма или рассылки из карточки объекта.
  - Метка времени (Timestamp with Timezone): Точный момент операции с учетом часового пояса (критично для логов и проводок).
  - Бизнес-номер (Sequence): Автогенерация номеров по шаблону (напр. ИНВ-2026-001).
  - Поле выбора цвета (Color Picker / Hex): Специфический тип для визуализации. В производстве и складской логистике цвета используются для маркировки статусов, категорий товаров или выделения приоритетных заказов на мониторе диспетчера.
- [ ] **Идентификация и Логистика:**
  - Штрих-код / QR-код Валидируемая строка с функцией генерации изображения для печати.
  - Серийный номер / Номер партии Поля для сквозной прослеживаемости от поставщика до продажи.
  - Геоданные (GIS/Coordinates) Широта/долгота для логистики и адресации.
- [ ] **Временны́е и Исторические (Temporal):**
  - Период (Date Range) Единый объект «Дата начала + Дата окончания» (для аренды, цен, отпусков).
  - Версионный тип (Temporal Data) Хранит историю изменений значения (позволяет увидеть, какая цена была год назад).
- [ ] **Финансовые и количественные типы:**
  - Денежный с валютой (Money/Currency) Связка «Число + Ссылка на валюту». Система блокирует прямое сложение разных валют без конвертации по курсу.
  - Единица измерения (Quantity with UOM) Число, жестко привязанное к классификатору (кг, шт, м³). Включает логику пересчета (напр. автоматический перевод из паллет в штуки при отгрузке).
- [ ] **Структурные и Организационные:**
  - Перечисление (Enum) Выбор из закрытого списка (Статусы, Типы).
  - Дерево/Иерархия (Hierarchy) Ссылка на «родителя» (для структуры цехов, групп товаров или должностей).
  - Табличная часть (Grid/Nested Table) Список строк внутри объекта (строки заказа, состав материалов).
  - Спецификация (BOM) Сложный «рецепт» изделия (компоненты + операции).
  - Расчетное поле (Formula / Calculated Field) Поле, значение которого не вводится вручную, а вычисляется автоматически на основе других полей. Например, «Сумма налога = Цена × Ставка НДС» или «Маржа = Цена продажи - Себестоимость». В ERP важно, чтобы эти расчеты происходили на уровне базы данных, а не только в интерфейсе.
- [ ] **Управление доступом и Логикой:**
  - Бизнес-роль (ACL) Настройка прав доступа к конкретному полю/кнопке.
  - Состояние (State Machine) Жесткая логика переходов объекта между статусами.
- [ ] **Типизированные вложения (Media & Docs):**
  - Фотография Изображения (товар, дефекты) с авто-превью.
  - Аудио / Видео Записи звонков, осмотров или процессов упаковки.
  - Документ (PDF/Office) Скан-копии с поддержкой версионности файлов.
- [ ] **Бизнес-объекты (Complex Structs)
  - Ссылка (Reference) Уникальный ID другого объекта системы.
  - Внешняя ссылка (External Link / URL) В отличие от системной ссылки (Reference), это ссылка на внешний ресурс. Ссылка на страницу товара на маркетплейсе, сайт производителя или внешний личный кабинет логистического оператора.
  - Сложный объект (Struct) Связка полей как единое целое (напр. Сумма + Валюта).
- [ ] **Массив сложных объектов
  - Список из нескольких сложных объектов (напр. "Несколько контактных лиц у поставщика", каждый со своими подполями).

### 6.3 Фильтрация и сортировка в списках
- [ ] **Базовая:** Фильтрация и сортировка по всем общим и стандартным полям сущности (доступна всем пользователям по умолчанию).
- [ ] **Расширенная (по спец. свойствам):** Администратор помечает конкретные динамические свойства как "доступные для фильтрации". Пользователь видит кнопку "Показать все фильтры" и может фильтровать список по специфическим полям категории (например, найти все трубы с Диаметром = 50мм).

### 6.4 Глобальные справочники и константы (Global Master Data)
- [ ] **Единые системные справочники:** Создание списков, доступных из любой точки системы (напр. «Валюты», «Языки», «Страны», «Единицы измерения»). При изменении в одном месте данные обновляются во всех связанных полях.
- [ ] **Глобальные константы (Примитивы):** Настройка фиксированных значений для всей компании (напр. «НДС = 20%», «Стандартная маржа = 15%», «Скидка для магазинов = 5%»). Используются в формулах и расчетах по умолчанию.
- [ ] **Автоматические фиды (Data Connectors):** Интерфейс для настройки онлайн-обновлений (напр. курсы валют из ЦБ, парсинг цен поставщиков по ссылке или API). Администратор сам сопоставляет внешние поля с полями системы.
- [ ] **Визуализация списков (Enums):** Настройка выпадающих списков с возможностью привязки иконок (флаги для языков) или цветовых меток (HEX-коды для статусов).
- [ ] **Правила пересчета (UOM Conversion):**
  - Матрица коэффициентов для автоматического перевода одних единиц в другие (напр. 1 паллета = 40 коробок) в масштабе всей системы.
  - Индивидуальные коэффициенты: Возможность задать специфический коэффициент пересчета для конкретной позиции номенклатуры (актуально для труб с разной толщиной стенки при одном диаметре), который приоритетнее глобального правила.
- [ ] **Редактор бизнес-логики:** Возможность администратору самому менять глобальные параметры без участия разработчика (напр. изменить ставку налога или добавить новый язык интерфейса одной кнопкой).

### 6.5 Принцип максимально простого UX
- [ ] **Zero-Code Admin Panel:** Администратор в специальном интерфейсе настраивает структуру любой сущности кликами и drag-and-drop — никаких технических знаний не требуется.
- [ ] **Простота для конечного пользователя:** Все настройки сложности (разделы, порядок, скрытые поля) — это забота администратора. Кладовщик и менеджер видят только чистый, понятный интерфейс без лишнего.
- [ ] **"Не повторить ошибки Excel":** Принцип проектирования — если действие требует более 2 кликов или объяснения, значит UX нужно упростить. Каждый экран должен быть интуитивным без обучения.
