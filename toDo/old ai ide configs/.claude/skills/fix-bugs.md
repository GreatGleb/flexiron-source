# Fix Bugs — Исправление багов

Один баг = один полный цикл. Не торопиться. Каждый шаг выполнять полностью перед следующим.

---

## CRITICAL RULES

1. **Read before touch** — прочитать ВЕСЬ файл перед правкой, не только строку с багом
2. **Verify the problem exists** — Grep или Read доказывает что проблема там есть, ПЕРЕД тем как что-то менять
3. **Plan before edit** — описать что именно изменится и почему, до того как открыть Edit
4. **Minimal change** — изменить ровно то что нужно. Не рефакторить вокруг, не "заодно улучшить"
5. **Check callers** — если меняется публичный API (функция, prop, тип) → grep всех, кто это использует
6. **Min 2 stops per bug** — СТОП-1 (после fix) + СТОП-2 (итог). Если цикл находит проблемы — каждая итерация добавляет стоп. Максимум 12.
7. **Verification cycle — max 10 iterations, stop on each error** — каждая итерация с ошибками = стоп: показать ошибки, спросить подтверждение перед fix. Итого 2 стопа минимум (чистая проверка), до 12 максимум (СТОП-1 + 10 итераций + СТОП-2).
8. **Mark ✅ только после завершения цикла проверки** — не после fix, не "скорее всего пройдёт"
9. **Never self-approve** — после fix не переходить к проверке без подтверждения пользователя
10. **Stop on each error iteration** — нашёл ошибки в цикле → показать ошибки → спросить подтверждение → только потом исправлять.

---

## Шаг 0: Инициализация (один раз)

```
1. Определить bugs-файл:
   - Найти toDo/plans/{plan}-*-plan.md (например 1.1-products-plan.md)
   - Bugs-файл: тот же prefix, каталог toDo/plans/bugs/, суффикс -bugs.md
   - Пример: "1.1" → toDo/plans/bugs/1.1-products-bugs.md

2. Прочитать bugs-файл целиком

3. Составить список незафиксированных:
   - "## БАГ-N" без "✅" → незафиксирован
   - "## ✅ БАГ-N" → уже исправлен, пропустить
   - "## PAT-N" → паттерн, не код — пропустить если не указан явно

4. Если аргумент bug указан → перейти к нему напрямую
   Если нет → показать список (формат ниже) и ждать выбора пользователя
```

**Формат списка:**
```
Незафиксированные баги — {BUGS_FILE}:

  БАГ-01 [Тип] — суть
  БАГ-05 [Runtime] — суть
  ...

Уже исправлено: ✅ БАГ-03, ✅ БАГ-07
PAT-* (паттерны, не код): PAT-01, PAT-02

С какого начать? (или "все по порядку")
```

## Цикл исправления одного бага

---

### Шаг А — Прочитать и понять баг полностью

1. Прочитать секцию бага в bugs-файле:
   - `**Файл:**` — список затронутых файлов
   - `**Серьёзность:**` — приоритет и контекст
   - `### Проблема` — что происходит, почему это баг
   - `### Исправление` — что написано как решение (может быть "TBD")

2. Прочитать КАЖДЫЙ файл из `**Файл:**` целиком (не только строку с багом):
   - Понять структуру вокруг проблемного места
   - Найти все места где проблема может проявляться

3. Если `### Исправление` = "TBD" или неполное:
   - Изучить проблему самостоятельно по `### Проблема`
   - Сформулировать план fix и ПОКАЗАТЬ пользователю перед применением
   - Ждать подтверждения

---

### Шаг Б — Верифицировать проблему

Перед любой правкой — доказать что проблема существует:

- **CSS-класс отсутствует** → Grep класса в импортируемых файлах компонента → убедиться что не найден
- **Неверный тип** → Read файл → найти строку с багом
- **Логическая ошибка** → Read функцию → проследить значение через код
- **Mock данные** → Read STORE → проверить отсутствие нужного поля
- **Неверный route name** → Grep name в router/index.ts → убедиться что не совпадает

Если проблема НЕ воспроизводится (уже исправлена ранее):
```
Проблема из БАГ-N не воспроизводится в текущем коде.
Возможно исправлена в другой сессии.
Отметить ✅ без правок?
```

---

### Шаг В — Спланировать fix

До открытия Edit — описать план:

```
Файл: src/...
Строка ~N: [что сейчас] → [что станет]
Причина: [почему это решает проблему]
Побочных эффектов: [нет / или: проверить X]
```

Если fix затрагивает публичный API (экспорт, prop, тип):
- Grep по всем потребителям (компоненты, composables, тесты)
- Перечислить в плане какие файлы ещё нужно изменить

---

### Шаг Г — Применить fix

Правила правки:
- Менять ровно то что описано в плане
- Не трогать соседний код (форматирование, порядок, стиль) если он не связан с багом
- Если баг в нескольких файлах → исправить все в рамках этого шага

После правки — прочитать изменённый фрагмент ещё раз и убедиться что выглядит правильно.

---

### СТОП-1 — После применения fix

Объявить пользователю что fix применён и запросить подтверждение на запуск проверки:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
БАГ-[N] — [название] — fix применён
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Изменения:
  [файл] строка ~N: [что было] → [что стало]

Намерен запустить цикл проверки (баг исправлен? → ТЗ → паттерны, до 10 итераций).
Запустить?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**Ждать подтверждения. Не запускать проверку до ответа пользователя.**

---

### Шаг Д — Цикл проверки (до 10 итераций)

Запускается только после подтверждения пользователя на СТОП-1.

Это **ручная проверка кодом** — не линтер, не автоматика. Claude читает файлы и анализирует.

Счётчик итераций начинается с 1.

**Каждая итерация — 3 уровня проверки по приоритету:**

---

**Уровень 1 — Баг исправлен? (главная проверка)**

Прочитать изменённые файлы и секцию `### Проблема` бага.
Задать себе вопрос: _это конкретное поведение теперь невозможно?_

- CSS-баг → найти класс в файле → убедиться что правило применяется к нужному селектору
- JS/TS баг → проследить логику от источника проблемы до результата
- Mock-баг → прочитать STORE → убедиться что нужное поле теперь есть и корректно
- Компонентный баг → найти место в template/script → убедиться что исправление активно

Если баг **НЕ исправлен** — найти причину:
- Не тот файл изменён? Не тот селектор? Не та переменная? Не тот компонент?
- Описать проблему в СТОП-итерация и предложить правильный path

---

**Уровень 2 — Соответствие ТЗ**

Сверить с `### Исправление` в bugs-файле:
- Все пункты исправления выполнены?
- Ничего не пропущено из описания?

---

**Уровень 3 — Паттерны проекта**

Проверить что fix не нарушает правила из `frontend_vue/CLAUDE.md` и `vue-rules.md`:
- Правильный способ использования компонента?
- Нет запрещённых паттернов (HTML-комментарии, span вместо input, router.back() и т.д.)?

---

**После всех 3 уровней:**

- **Всё чисто** → выйти из цикла → перейти к Шагу Е
- **Найдены проблемы, итерация < 10** → СТОП-итерация (формат ниже) → ждать подтверждения → исправить → счётчик +1 → повторить итерацию
- **Итерация = 10, проблемы остались** → выйти из цикла → Шаг Е → в СТОП-2 указать лимит

**Формат СТОП-итерации (при проблемах):**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Итерация [N]/10 — найдены проблемы
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[Уровень 1 / 2 / 3]:
  [описание проблемы]
  [что именно не так: файл, строка, причина]

Намерен исправить: [план одной строкой]
Продолжить?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

### Шаг Е — Проверить смежный код (если нужно)

Только для багов типов CSS, Design, Runtime — быстрая проверка что fix не сломал соседнее:

- **CSS-баг** → Grep изменённого класса по всем другим компонентам → не сломали глобально?
- **Runtime-баг в composable** → Grep функции по views → проверить что все callers получают то что ожидают
- **Mock-баг** → Grep изменённого ID/поля → не сломали другие mock функции?

Если явно ничего смежного нет — пропустить этот шаг.

---

### Шаг Ж — Отметить ✅ в bugs-файле

Два изменения в bugs-файле:

**1. Заголовок секции бага:**
```
## БАГ-N — [название]
         ↓
## ✅ БАГ-N — [название]
```

**2. Строка в итоговой таблице:**
```
| БАГ-N | Тип | файл | суть |
         ↓
| ✅ БАГ-N | Тип | файл | суть |
```

---

## Завершение

После закрытия всех багов — запустить typecheck + lint один раз:

```bash
cd frontend_vue && npm run typecheck && npm run lint
```

**Если ошибки найдены:**
- Прочитать каждую ошибку
- Определить в каком файле и связана ли с одним из исправленных багов
- Исправить — по тем же правилам: Read → план → Edit → прочитать результат
- Запустить повторно до 0 ошибок
- Если ошибка явно не связана с исправленными багами — сообщить пользователю отдельно, не исправлять молча

**Если ошибок нет:**


## Особые случаи

### Исправление "TBD"
`### Исправление` написано "TBD" → самостоятельно изучить `### Проблема` → сформулировать fix → **показать пользователю и ждать подтверждения** перед правкой.

### Несколько файлов в одном баге
Исправить все в рамках одного цикла → одна пометка ✅.

### PAT-* (паттерны)
Паттерны описывают правила на будущее — не требуют правки кода. Пропустить автоматически. Если пользователь указал PAT-N явно — объяснить: "Это паттерн, не баг. Код не менялся намеренно."

### Баг в нескольких несвязанных местах
Если `**Файл:**` содержит файлы с несвязанной логикой (например и composable, и CSS) — исправить оба в одном цикле, одна пометка ✅.

### Регрессия после финального typecheck
Если при запуске typecheck + lint в Завершении появились ошибки в файлах НЕ из исправленных багов:
- Прочитать ошибки
- Если связаны с одним из fix → исправить
- Если не связаны → сообщить пользователю отдельно, не исправлять молча
