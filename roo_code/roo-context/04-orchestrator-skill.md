# План: Оркестратор-исполнитель архитектура для create-page

> Источник: [`orchestrator-executor-skill.md`](../toDo/orchestrator-executor-skill.md)

## Идея

Переписать `/create-page` на двух-агентную модель:

- **Agent 1 (оркестратор)** — читает план, генерирует промпт для одной фазы, запускает Agent 2, получает результат, генерирует следующий промпт
- **Agent 2 (исполнитель)** — получает один промпт, выполняет одну фазу, возвращает summary

## Почему это лучше текущего подхода

Сейчас при выполнении 10 фаз в одном чате я знаю что впереди ещё 9 фаз — это создаёт контекстное давление и торопливость.

В новой архитектуре каждый Agent 2 стартует с чистым контекстом только своей фазы. Нет давления "мне ещё X фаз впереди". Качество генерации каждой фазы выше.

Это достигается без STOP и без "ок" от пользователя — процесс полностью автономный.

## Известное ограничение

Agent 2 для фазы N должен знать что написали предыдущие фазы. Agent 1 передаёт summary предыдущих фаз в каждый следующий промпт — и сам накапливает контекст. Но накопление у оркестратора (который только планирует), а не у исполнителя (который пишет код). Исполнитель каждый раз чистый.
