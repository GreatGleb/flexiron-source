## Блоки работ до MVP (в порядке приоритета)

### 1. Settings — Admin Config (`page Настройки`)
**Почему first:** без настроек нельзя управлять налогами, валютами, единицами измерения, статусами, пользователями.

**Состав:**
- Компания (реквизиты)
- Финансы (PVM ставка, маржа, валюты)
- Единицы измерения (UOM + матрица пересчёта)
- Статусы заказов (color picker, порядок)
- Карта склада (секторы)
- Пользователи (CRUD + роли)

**Детальный план:** [`settings-plan.md`](roo_code/plans/general/settings-plan.md)

---

### 2. Финансы (`page Финансы`) — MVP-упрощение
Не дашборд с дебиторкой, а **лента всех финансовых операций**:

| Элемент | Описание |
|---------|----------|
| **Лента операций** | Таблица-хронология: дата, тип, сумма, контрагент, статус |
| **Типы операций** | Оплата от клиента (in), оплата поставщику (out), ручная запись |
| **Ссылка на источник** | Из строки операции — переход в карточку заказа/поставщика |
| **Ручной ввод** | Кнопка "Добавить операцию" → модалка: тип, сумма, дата, контрагент, комментарий |

**Не входит в MVP Finance:**
- Дебиторка / аналитика задолженностей
- Типы оплат как справочник
- Частичные оплаты с индикатором
- Возвраты
- Напоминания

---

### 3. Генерация документов (встроено в `page Заказы`)
Feature flag `orderDocumentGen = true`, но реализация — заглушка.

| Подзадача | Описание |
|-----------|----------|
| 3.1 Логика определения пакета | Local/Export + Самовывоз/Доставка → предлагаемый набор документов |
| 3.2 Предпросмотр | Редактирование перед генерацией (вес, цены, реквизиты) |
| 3.3 Генерация PDF | Invoice, CMR, KPO, Сертификаты, Lipdukai |
| 3.4 Архив документов | Сохранение PDF в истории заказа, версионность |

---

### 4. Связь Заказа со Складом (Orders ↔ Warehouse)
**Критическая интеграция:** сейчас заказ и склад — независимые сущности.

| Подзадача | Описание |
|-----------|----------|
| 4.1 Резервирование товара | При создании заказа — блокировка stock на складе |
| 4.2 Списание при отгрузке | При статусе "Shipped" — окончательное списание |
| 4.3 Отрицательные остатки | Продажа без наличия → остаток в минус → сигнал закупщику |
| 4.4 Механизм "Отрезать" (Cutting) | Создание обрезка (offcut) из заказа |

---

### 5. Заказы поставщикам (Purchase Orders)
Новая сущность со своим lifecycle.

| Подзадача | Описание |
|-----------|----------|
| 5.1 Сущность PurchaseOrder | Жизненный цикл: draft → sent → confirmed → in_transit → completed |
| 5.2 Страница списка PO | Таблица с фильтрацией по статусу, поставщику, датам |
| 5.3 Карточка PO | Шапка, товары, статус, приёмка партий |
| 5.4 Связь с дефицитом | Из дефицита → создать PO, при приёмке → закрыть дефицит |
| 5.5 Связь с Movements | При создании Movement receipt → reference на PO |

---

### 6. Доработки CRM (баги и мелкий функционал)
Из [`2-crm-analysis-plan.md`](toDo/plans/2-crm-analysis-plan.md):

| Подзадача | Описание |
|-----------|----------|
| 6.1 Inline-редактирование товаров | quantity/price/discount в таблице заказа (API поддерживает) |
| 6.2 Отображение НДС и скидок | totalVat, totalWithVat, col_discount — данные есть, UI нет |
| 6.3 Маржинальность услуг | col_cost + col_margin в таблице услуг |
| 6.4 История заказов в клиенте | `orderHistory` приходит, UI не показывает |
| 6.5 Валидация email/companyCode | ClientCreatePage — required поля |
| 6.6 Канбан-доска | `orderKanbanView = false` — включить и реализовать |

## Рекомендуемый порядок реализации

```
1. Settings (Admin Config)      — без него нельзя настраивать систему
2. Финансы (лента операций)     — упрощённая страница
3. Генерация документов         — критично для продаж
4. Связь Заказа со Складом     — резервирование и списание
5. Purchase Orders              — закупки
6. Доработки CRM                — баги и UX
7. Физический склад             — post-MVP
```
