# Update Skills — Анализ корня и обновление скилов

После исправления багов — найти ПРИЧИНУ почему скил допустил каждый баг, и закрыть именно эту дыру.

**Принцип:** не копировать "правило на будущее" из бага в скил. Вместо этого — понять что именно в скиле было написано недостаточно подробно, слишком коротко, или вообще отсутствовало — и дописать именно там.

---

## CRITICAL RULES

1. **Root cause first** — перед любой правкой ответить на вопрос: «В каком месте какого скила и что именно позволило этому багу случиться?»
2. **Найти пробел, не симптом** — если баг «мало данных», причина не «добавь больше данных», а «скил не указывал минимальное количество для данного случая»
3. **Точное место** — изменение вносится в конкретную фазу/секцию, где разработчик принимал бы решение — не в конец файла
4. **Не дублировать** — Grep перед вставкой
5. **Только ✅ исправленные** — незафиксированные баги не трогать
6. **One bug per stop**

## Шаг 0: Инициализация

```
1. Определить bugs-файл:
   - toDo/plans/{plan}-*-plan.md → toDo/plans/bugs/{plan}-*-bugs.md

2. Прочитать bugs-файл целиком

3. Собрать список:
   - Только "## ✅ БАГ-N"
   - PAT-* — пропустить

4. Прочитать скилы которые могут обновляться:
   - .claude/skills/create-page.md — полностью
   - .claude/skills/create-plan.md — полностью (если существует)
   - .claude/skills/vue-rules.md — полностью

5. Показать план анализа (формат ниже) → ждать подтверждения на старт
```

**Формат плана:**
```

  БАГ-01 — [суть]
  БАГ-05 — [суть]
  ...

Пропущено (PAT-*): PAT-01, PAT-02

Начать анализ?
```

## Цикл анализа одного бага

---

### Шаг А — Прочитать баг

Из секции бага:
- `### Проблема` — что произошло
- `### Исправление` — как исправили
- `### Правило на будущее` — что было написано как урок (отправная точка, не финальный ответ)

---

### Шаг Б — Анализ корня: почему скил это допустил?

Это самый важный шаг. Задать себе вопросы:

**1. Какое решение принималось когда этот баг был совершён?**
- На какой фазе реализации страницы?
- Что писал разработчик в тот момент (mock данные? шаблон? CSS? composable?)?

**2. Планировочный пробел или пробел реализации?**

Это ключевой вопрос — от него зависит целевой скил:

- **Планировочный пробел** → баг мог быть предотвращён если план явно указал требование.
  Признаки: план не указал структуру данных, объём mock, список полей, связи между сущностями, обязательные категории.
  → Целевой скил: `create-plan.md`

- **Пробел реализации** → требование было понятно из контекста, но скил создания страницы не описал как правильно это реализовать.
  Признаки: CSS паттерн, структура composable, контракт v-model, порядок шагов.
  → Целевой скил: `create-page.md`

```
Примеры:
  Баг "cat-2 не имеет inherited fields в mock"
  → План должен был указать: "для каждой категории перечислить inheritedFields и включить их в mock данные продуктов"
  → Это планировочный пробел → create-plan.md

  Баг ".empty-state не определён в CSS карточки"
  → Требование было очевидно (класс используется в шаблоне), но скил не описал что .empty-state не глобальный
  → Это пробел реализации → create-page.md Phase 7
```

**3. Что было в целевом скиле в этом месте?**
- Прочитать соответствующую секцию целевого скила
- Была ли там инструкция которая могла предотвратить этот баг?
- Если была — почему не помогла: слишком коротко, без примера, без явного требования?
- Если не было — значит это пробел

**4. Сформулировать корень:**

```
Баг случился потому что:
  В [скил], [фаза/секция] — [что именно отсутствовало или было недостаточно описано]
```

**5. Определить целевой скил и место:**

| Тип пробела | Целевой скил | Место |
|---|---|---|
| Требование к структуре данных не указано в плане | `create-plan.md` | секция структуры данных / mock |
| Объём или состав mock не указан в плане | `create-plan.md` | секция mock данных |
| Связи между сущностями не описаны в плане | `create-plan.md` | нужная секция |
| Любой другой планировочный пробел | `create-plan.md` | нужная секция |
| Паттерн написания mock данных | `create-page.md` | Phase 2: Mock Data |
| Паттерн написания сервиса | `create-page.md` | Phase 3: Service Layer |
| Паттерн написания composable | `create-page.md` | Phase 4: Composable |
| Паттерн написания переводов | `create-page.md` | Phase 5: i18n |
| Паттерн написания шаблона/CSS | `create-page.md` | Phase 6 или 7 |
| Паттерн регистрации роутов/флагов | `create-page.md` | Phase 8 |
| Паттерн финальной проверки | `create-page.md` | Phase 9: Verification |
| Повторяемый паттерн любой страницы | `vue-rules.md` | новый питфолл в конце |
| Специфично для одной страницы | нигде — оставить в bugs-файле |

### Шаг В — Проверить что пробела ещё нет

Grep ключевых слов в целевом файле и секции:
- Нашёл достаточно подробное правило → **не дублировать**, объяснить почему пропускаем
- Нашёл упоминание но слишком кратко → предложить расширить существующее место
- Не нашёл → вставить новое

---

### Шаг Г — Сформулировать текст для вставки

**Для create-page.md или create-plan.md:**

Текст должен быть конкретным и actionable — не общим советом:

```
❌ Плохо: "Добавляй реалистичные данные"
✅ Хорошо: "Для таблиц с пагинацией — минимум 50 записей в STORE.
            Меньше = пагинация не активируется, scroll не тестируется."

❌ Плохо: "Проверяй CSS классы"
✅ Хорошо: "`.empty-state` НЕ глобальный — не входит в admin-core.scss.
            Если компонент использует `.empty-state` — определить его в своём page CSS."
```

Формат вставки — встраивать в существующий список или текст фазы, не добавлять новый заголовок:
```markdown
- **[Конкретное требование]** — [почему именно так, последствие если нарушить]
```

**Для vue-rules.md (новый питфолл):**

```markdown
### [N]. [Конкретное название]

[Описание ситуации — что происходит и почему неочевидно]

**Fix:** [конкретное решение]

```пример если нужен```
```

Номер = последний существующий + 1.

## Особые случаи

**Баг слишком специфичен для данной страницы:**
Например "inherited fields для cat-2 в products mock" — это не паттерн, это данные. В скилы не идёт. Сообщить: "Специфично для домена Products, не добавляю в скилы."

**Место вставки неочевидно:**
Показать два варианта с аргументами → спросить пользователя.

**Правило уже частично есть, но недостаточно подробно:**
Предложить расширить существующий текст, не добавлять новый блок. Показать до/после.
