# Add Bug — Запись ручного бага в файл

Пользователь описывает найденный баг в свободной форме. Задача — правильно оформить и записать в bugs-файл.

---

## Шаги

**1. Определить bugs-файл:**
- Найти файл плана: `toDo/plans/{plan}-*-plan.md` (например `1.1-products-plan.md`)
- Bugs-файл: то же имя, каталог `toDo/plans/bugs/`, `-plan.md` → `-bugs.md`
- Прочитать итоговую таблицу файла → определить NEXT_BUG_ID

**2. Прочитать описание пользователя:**
- Пользователь пишет что нашёл — в свободной форме, на любом языке
- Если из описания неясно: файл, серьёзность или конкретное проявление — задать один уточняющий вопрос (не несколько)

**3. Оформить и записать баг:**

Формат записи:

```markdown
## БАГ-[N] — [Краткое название]

**Файл:** `[файл или компонент, если известен]`  
**Серьёзность:** Высокая / Средняя / Низкая — [одна фраза почему]

### Проблема

[Что происходит — конкретно, как пользователь видит баг]

### Исправление

[Что нужно сделать — если очевидно из описания; иначе: "TBD"]

### Правило на будущее

[Как избежать — если применимо; иначе опустить секцию]
```

Добавить строку в итоговую таблицу в конце файла:
```
| БАГ-[N] | [Тип] | `[файл]` | [Суть] |
```

**4. Подтвердить:**

```
✅ БАГ-[N] записан в [путь к bugs-файлу]
[Краткое название как подтверждение]
```

---

## Типы багов для таблицы

| Тип | Когда |
|---|---|
| Runtime | Ошибка в поведении во время работы |
| TypeScript | Ошибка типов |
| CSS | Визуальная ошибка, неверные стили |
| Design | Несоответствие дизайн-паттернам проекта |
| Mock data | Неверные данные в mock STORE |
| i18n | Проблема с переводами |
| Code style | Нарушение правил кода (комментарии, паттерны) |
| UX | Неудобное или неожиданное поведение для пользователя |
